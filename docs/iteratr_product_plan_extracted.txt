1. Product Vision
The Elevator Pitch
iteratr is an autonomous, full-stack coding mentor that replicates the experience of pair-programming
with a senior engineer. It dynamically scales problem difficulty using an Elo rating system, securely
executes user code to analyze logic failures, and uses real-time multi-agent panels to conduct mock
technical interviews — all while never just telling the user they are wrong.
1.1 The Core Problem
Every major coding platform today — LeetCode, HackerRank, Codewars — has the same
fundamental flaw: they tell you if you are right or wrong, then show you the answer. This creates
passive learning. The user reads the solution, thinks they understand it, then fails the same type
of question two days later.
iteratr fixes this with three principles:
• Never give the answer directly — always guide with Socratic hints
• Personalize the difficulty dynamically — not static Easy/Medium/Hard tracks
• Teach through doing and failing — real code execution with real feedback

companies.

3. Phase 1 — Core Testing Engine
Goal: Build the minimum viable product. A user should be able to log in,
take a multi-type test, get hints when wrong, and see their result.
3.1 Multi-Type Question Engine
The question engine is the heart of iteratr. Every question is generated by the AI into a strict
JSON schema. There are four question types, each testing a different cognitive layer.
Type 1 — Multiple Choice (MCQ)
Tests conceptual understanding with four options. The wrong answers (distractors) are
generated from real misconceptions, not random noise. The AI is explicitly prompted to ask:
"What would a student who half-understands this concept think?" This makes MCQs genuinely
educational.
• 4 options: 1 correct, 1 clearly wrong, 2 plausible distractors
• Distractors map to known misconceptions (e.g. O(log n) for hash maps because
students confuse them with BSTs)
• On wrong answer: show which option was correct AND explain why the chosen option
fails
• Hint: never say "wrong" — say "here is why that reasoning breaks down"
• Elo update: +12 for correct first try, +6 for correct second try, -8 for fail
Type 2 — Fill in the Blank
Tests the blank that matters — not syntax trivia, but the conceptual term or value that reveals
whether the student truly understands. The AI chooses the blank strategically: it picks the word
whose absence forces active recall of the core concept.
• 1 to 3 blanks per question depending on complexity
• Blank is always the conceptual gap, not a random word
• Partial credit: each blank scored independently
• Hint: if wrong, describe what the blank represents without giving the answer
• Example: "A recursive function must have a ___ case to stop and a ___ case to call
itself"

The most powerful question type. The user writes real code in a Monaco Editor (VS Code
engine). The code is sent to a secure sandboxed execution environment (Judge0 API) and run
against hidden test cases. The AI agent then reasons about the failure and generates a
personalized hint.
• Languages: Python, C++, JavaScript, TypeScript (Phase 1 starts with Python only)
• Partial scaffolding provided: function signature + comments, never a blank screen
• Code runs against 5 to 10 hidden test cases including edge cases
• On failure: agent identifies which test case failed and WHY (empty array, negative input,
etc.)
• Hint system: 4 progressive levels (see Section 5 for full hint engine spec)
• Elo factors in: correctness, time taken, and number of hints used
Type 4 — Drag to Order
Tests procedural understanding — knowing the correct sequence of steps. Perfect for
algorithms (binary search, merge sort), system design flows (what happens when you press
Enter in a browser), or CS concepts (TCP handshake, OS boot sequence). Drag and drop, no
typing required.
• 4 to 8 steps presented in shuffled order
• User drags to arrange in correct sequence
• On wrong order: show first incorrect step and hint about what must come before it
• Great warm-up question before a harder code space challenge on the same topic
3.2 Live Code Execution (Sandbox)
User code is NEVER executed directly on the iteratr server. All code runs in a secure, isolated
Docker container via the Judge0 API or Piston. This protects the server from malicious code
while providing real execution results.
Component Implementation detail
Execution service Judge0 API (via RapidAPI) in Phase 1. Self-hosted Piston in Phase
3+ for cost control.
Supported languages Python 3.11 (Phase 1), C++ 17, JavaScript Node 20 (Phase 2)
Timeout 5 seconds per execution. Infinite loops are killed automatically.
Hidden test cases 5 to 10 per question. Stored server-side, never exposed to the
client.
Output parsed stdout, stderr, exit code, execution time, memory used
Edge cases tested Empty input, null, negative numbers, max integer, single element
arrays

4. Phase 2 — Adaptive Intelligence
Goal: Make the platform feel intelligent. Difficulty adjusts to the user.
The AI knows what they struggle with.
4.1 Elo Rating System
Every user has a hidden Elo rating (starting at 1200). Every question in the database also has
an Elo rating. The system works exactly like chess matchmaking: solve a hard question quickly
and your rating spikes; fail an easy one and it drops. The next question served always matches
the user's current rating.
Outcome Elo change
Correct, 0 hints, fast +18 to +24 (depends on question Elo vs user Elo)
Correct, 1 hint used +10 to +14
Correct, 2+ hints used +4 to +8
Incorrect after 4 hints -6 to -12
Time limit exceeded -4 to -8
Correct on second attempt +6
Question difficulty bands:
• 800–1000: Foundations (basic loops, arrays, string manipulation)
• 1000–1200: Core DSA (stacks, queues, linked lists, basic trees)
• 1200–1400: Intermediate (BFS, DFS, binary search, recursion, hashmaps)
• 1400–1600: Advanced (dynamic programming, graphs, backtracking)
• 1600–1800: Expert (system design components, complex DP, segment trees)
• 1800+: FAANG-level (hard LeetCode equivalents, optimization problems)
4.2 Knowledge Gap Detector
The AI tracks every failure at the sub-topic level — not just "Arrays" but "Two-pointer
technique", "Sliding window", "Array rotation". If a user fails 3 or more questions in a sub-topic,
the system flags it as a weak zone.
• Weak zones shown as red on the topic mastery map
• System automatically routes 2 easier foundational questions on the weak topic before
advancing

5. The Socratic Hint Engine (Core Differentiator)
This is the feature that makes iteratr different from every other platform. When a user gets an
answer wrong, iteratr never says 'Incorrect. The answer is X.' Instead, the AI reasons about
WHY the user was wrong and delivers a precisely calibrated hint.
Design principle: A hint should close the gap between what the user
knows and what they need to know — without bypassing the thinking.
5.1 Hint Levels
Level What the hint does
Level 1 — Concept gap Explains WHY the submitted answer is logically wrong. Does not
point toward the correct answer. Example: 'O(log n) is for binary
search trees where you halve the search space. A hash map does
not search — it computes.'
Level 2 — Directional nudge Points toward the general concept without naming it. Example:
'Think about what operation a hash map performs that makes
searching unnecessary.'
Level 3 — Structural scaffold For code questions: gives pseudocode structure without working
syntax. Example: 'You need: a pointer for prev (starts None), a
pointer for curr, and a way to save next before you overwrite it.'
Level 4 — Full walkthrough Complete explanation with annotated code. Awarded only after 3
failed attempts. Also flags the topic as a weak zone in the user's
profile and applies a small Elo penalty.
5.2 The ReAct Agent Loop (Code Questions)
For code space questions, the hint is not just looked up from a database. A ReAct (Reason +
Act) agent actively analyzes the failure:
1. User submits code.
2. Agent Reason: 'I need to run this against hidden test cases.'
3. Agent Action: Calls ExecuteCodeTool — sends code to Judge0 sandbox.
4. Observation: Code failed on test case 3 (empty array input).
5. Agent Reason: 'User missed the empty array edge case. They should not be told
directly. I will hint about edge case handling.'
6. Agent Output: Level 1 hint targeting the specific gap identified.

6. Phase 3 — Mock Interview Suite
Goal: Make iteratr the place developers go the week before their
interview.
6.1 AI Interview Panel
A real-time mock interview experience powered by two AI agents running in parallel via
WebSocket connections.
Agent Role
Agent 1 — The Interviewer Conducts the interview in real time. Asks DSA or system design
questions, responds to the user's answers, asks follow-up
questions, simulates interruptions and clarifying questions. Fully
conversational.
Agent 2 — The Silent Grader Runs invisibly in the background. Monitors the user's code quality,
response latency, communication clarity, and problem-solving
approach. Writes a score to the database at the end of the session.
Never speaks during the interview.
6.2 Post-Interview Scorecard
After every mock interview, the user receives a detailed scorecard. This is the output they can
share, save, and use to track improvement over time.
• Code quality score (0–100): correctness, time complexity awareness, edge case
handling
• Communication score (0–100): clarity of explanation, asks good clarifying questions,
thinks out loud
• Speed score (0–100): time to first approach, time to working solution
• Hint dependency score: how many hints were needed, at which stages
• Full transcript: every message and code submission with AI commentary
• "What you should have said/coded differently": specific, actionable feedback on each
section
• Overall interview grade: Pass / Strong Pass / No Hire / Strong No Hire

7. Phase 4 — Analytics & Growth
Goal: Make progress visible. Give users a reason to come back every

7.1 Skill Heatmap
A GitHub-style activity grid combined with a topic coverage map. The heatmap shows daily
activity. The topic map shows which subjects are strong (green), weak (red), or untouched
(gray). This is the dashboard users will screenshot and share.
• Activity grid: 52-week view of daily question attempts
• Streak counter: current streak and longest streak prominently displayed
• Topic coverage radar: all major DSA topics shown with fill level (0-100%)
• Elo history line chart: rating over the past 30 days with milestone markers
• Weak zone callouts: top 3 areas to focus on this week
7.2 Daily Challenge & Streak System
One new question every day, matched to the user's current Elo. Solving it extends the streak.
Missing a day does not break the streak — but it pauses the daily bonus multiplier.
• Daily challenge unlocks at midnight local time
• Streak bonus: consecutive days multiply Elo gains by 1.1x, 1.2x, up to 1.5x
• Grace period: one streak freeze per week if the user misses a day
• Weekly challenge: harder question released every Monday, bonus Elo on completion
7.3 Shareable Public Profile
Every user gets a public profile page at iteratr.online/u/[username]. This page is designed to be
put on a resume or LinkedIn profile.
• Elo rating displayed prominently with percentile (e.g. Top 12%)
• Topics mastered: list of sub-topics where the user has solved 5+ questions without hints
• Mock interview best score per company mode
• Activity streak and total questions solved
• "Interview Ready" badge unlocked after scoring 80+ on a FAANG mode mock interview
